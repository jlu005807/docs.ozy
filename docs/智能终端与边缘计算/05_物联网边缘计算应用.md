# 物联网边缘计算应用

## 边缘计算应用的总体架构 

在实际的物联网（IoT）应用中，网络边缘是一个相对的概念，架构也随着需求在不断演进。

1. 两类边缘与六大演进架构

| 架构类型           | 核心特点与定义                                               | 考试速记“题眼”                                               |
| ------------------ | ------------------------------------------------------------ | ------------------------------------------------------------ |
| **两类边缘**       | 根据实际网络架构，分为**主干网边缘**和**泛在边缘**。         | 看到“边缘分类”，对应“主干网”与“泛在”。                       |
| **三层“云-边-端”** | 包含云中心层、边缘层以及前端层。计算请求根据数据量大小（KB到PB级）流向不同层级。 | 典型协同：边缘IaaS与云端IaaS资源协同；边缘PaaS与云端PaaS智能协同。 |
| **两层“边-端”**    | 随着应用对延迟敏感度极度提高，**边缘网络独立承担终端设备的服务请求**，云被“隐去”。 | 看到“独立承担请求/延迟极度敏感”，选“边-端”两层架构。         |
| **多接入边缘计算** | 应对超密集网络，每个前端设备可以**同时接入多个边缘服务器**。 | 关键词：**超密集网络**、**接入多个**。                       |
| **分布式D2D/D4D**  | Device-to-Device/Device-for-Device，边缘设备通过直连方式**相互成为其他设备的边缘计算节点**。 | 看到设备之间**相互作为计算节点**，选 D2D/D4D。               |
| **AIoT 架构**      | 人工智能（AI）+ 物联网（IoT）。在系统层面，**AI的能力是通过边缘计算来提供的**。 | 边缘计算是AIoT中“AI”落地的真正载体。                         |

## 基于MEC的CDN系统

**为什么需要边缘CDN？** 用户需求增长最快的是视频类业务。高清直播需>50Mbps，多路并发>200Mbps。仅靠云数据中心会造成极大的网络拥堵和延迟。 

**解决方案**：采用**IaaS方式**在电信边缘云设置虚拟CDN节点。

边缘CDN基于现有CDN系统提供的历史数据，结合移动通信网在不同位置的网络流量、用户内容需求，提前将用户可能需要的内容缓存到相应的边 缘CDN节点。

### 边缘CDN工作流程

请记住，整个过程的核心中转站是**“DNS代理”**：

- **发出请求**：移动终端向**DNS代理**发出DNS查询请求。
- **首次转发**：DNS代理将请求转发给**CDN的DNS服务器**。
- **返回CNAME**：CDN的DNS服务器向DNS代理返回一个**DNS响应（CNAME）**。
- **二次转发**：DNS代理将请求转发给**CDN全局负载均衡器**。
- **返回IP(A记录)**：负载均衡器根据网络拓扑和负载，向DNS代理返回最适合的边缘服务器IP地址，即**A记录**。
- **下发A记录**：DNS代理将A记录传给用户终端。
- **建立连接**：用户终端根据A记录，与指定的边缘服务器建立连接获取视频。

## 三大典型应用场景

本节展示了三个非常具体的应用场景，重点考察边缘计算在其中扮演的“特定角色”。

1. 基于MEC的增强现实（AR/VR）服务

- 基于MEC的实时目标跟踪（安防监控）

- 基于MEC的智能家居系统

**痛点**：通信协议极度碎片化（Wi-Fi, ZigBee, 蓝牙等），接入网络五花八门（宽带, 4G/5G, 有线电视网）。

**智能家庭网关的角色（底层）**：负责**不同网络通信协议的识别与转换**、动态带宽调整、定向管道加速。

**边缘云的用途（高层桥梁）**：它是连接家居设备与云数据中心、居住者、开发者的**桥梁**，提供编程接口、数据管理、安全隐私保护和设备控制。
